================================================================================

Q1: KEEP YOUR CUSTOM RENDERER
─────────────────────────────────────────────────────────────────────────────
✓ You already have the optimal setup: custom WebGPU with MRT (color+motion)
✓ Three.js/Babylon.js add unnecessary abstraction layers
✓ Physics integration is just: add Cannon.js separately + hook up transforms
✓ Your motion vector computation is production-ready

Q2: INSTANCED RENDERING FOR 100 BODIES
─────────────────────────────────────────────────────────────────────────────
✓ Use storage buffer with InstanceData struct (144 bytes per body)
✓ Struct: [16 floats model][16 floats prevModel][4 floats color]
✓ Single draw call: draw(vertexCount, 100)
✓ Vertex shader indexes buffer via @builtin(instance_index)
✓ Results in ~5-10x speedup vs individual draw calls
✓ ~14 KB memory for 100 bodies (negligible on modern GPU)

Q3: MOTION VECTORS FROM PHYSICS
─────────────────────────────────────────────────────────────────────────────
✓ Store prevModelMatrix per physics body (updated each frame)
✓ Your existing shader already computes: (currNDC - prevNDC)
✓ This automatically includes physics body motion + camera motion
✓ Formula: motion = (currentWorldPos - previousWorldPos) projected to screen
✓ Pattern: Save current frame's transforms → use as previous next frame

Q4: GEOMETRY FOR DOMINOES/PENDULUM
─────────────────────────────────────────────────────────────────────────────
✓ Box: Reuse existing cubeVertices(), scale via transform matrix
✓ Sphere: UV sphere with 16 segments → ~289 vertices (efficient)
✓ Cylinder: Capped cylinder with 8 segments → ~80 vertices
✓ Don't create geometry variants; transform + instancing is more efficient
✓ Sphere code example provided in QUICK_REFERENCE.md

Q5: CAMERA-RELATIVE MOTION VECTORS
─────────────────────────────────────────────────────────────────────────────
✓ Your code already does this correctly!
✓ Use prevViewProj from previous frame's camera matrices
✓ Formula: prevNDC = prevViewProj × prevWorldPos
✓ Divide by W for perspective correction (automatic in interpolation)
✓ Result: motion = object_motion - camera_motion (relative)
