# Web Demo V3 — Physics + Noise Warp

## Goal
A domino-falling demo where dominoes topple in a chain reaction, knocked over by a pendulum on a string that the user controls. The noise warp visualizes temporally coherent noise over the physics scene — same algorithm as V2, but now with many independently moving rigid bodies instead of a single spinning cube.

## Why This Demo
The spinning cube in V2 demonstrates noise warping but is trivial motion. Dominoes test the algorithm under **complex, chaotic, multi-body motion** — dozens of objects moving independently with collisions, rotations, and sudden velocity changes. This stress-tests the warp's ability to maintain temporal coherence when the motion field is highly non-smooth.

---

## Key Decisions

### Physics Engine: Rapier.js
**Why Rapier over alternatives:**
- **Performance**: Rust/WASM, 2-5x faster than pure JS alternatives. 100 bodies is trivial (benchmarked at 120fps with 4500 bodies).
- **Joints**: SphericalJoint for pendulum, perfect API: `JointData.spherical(anchorA, anchorB)`.
- **Friction**: Per-collider friction coefficients with combine rules. Critical for realistic domino cascades (friction 0.25-0.4 optimal).
- **Active**: Dimforge maintains it, 2025 had SIMD + BVH improvements.
- **Size**: ~1.4MB WASM, acceptable for a demo.

Rejected:
- **Cannon-es**: Adequate for 100 bodies but pure JS = slower. No WASM advantage.
- **Jolt**: Most features (soft-body, layers) but C++-centric docs, overkill for rigid-body-only demo.
- **Ammo.js**: Unmaintained, complex API.

### Rendering: Raw WebGPU (reuse V2 pipeline)
**Why not Three.js:**
- We already have a complete WebGPU pipeline with MRT (color + motion vectors), compute shaders, and noise warp.
- Three.js WebGPU requires TSL node materials — no raw ShaderMaterial, can't write custom WGSL.
- Three.js doesn't provide native motion vectors. We'd have to hack them in anyway.
- Adding Three.js (~600KB) to render boxes and a floor is overkill when we already do this in ~200 lines of WGSL.

What changes from V2: instanced rendering. V2 draws 1 cube + 1 floor. V3 draws N dominoes + 1 floor + 1 pendulum ball + 1 string. Use a single instanced draw call with a storage buffer of per-instance transforms (model matrix + prev model matrix for motion vectors).

### UI Framework: Vanilla JS (no Svelte)
**Why:** V2 already works with vanilla JS. The complexity is in GPU compute, not UI. Good file separation achieves the same modularity goals without a build tool. We can add Svelte later if UI grows further.

**How we achieve separation of concerns:** Split V2's monolithic files into focused modules (see file structure below).

## Architecture

### Data Flow (per frame)
```
1. Physics step (Rapier WASM, CPU)
   ↓ copy transforms to storage buffer
2. GPU scene render (instanced MRT: color + motion vectors + depth)
   ↓
3. GPU compute: build deformation → clear → backward map → brownian bridge → normalize
   ↓
4. [optional] GPU compute: blue noise post-process
   ↓
5. GPU display pass
```

Step 1 is the only CPU work beyond command buffer submission. Rapier runs on CPU (WASM), outputs position + quaternion per body. We pack these into a GPU storage buffer of mat4×2 (current + previous model matrices) and upload once per frame.

### Instanced Rendering for Motion Vectors
V2 uses uniform buffers for a single model matrix + prevModel matrix. V3 needs N model matrices.

**Approach**: Storage buffer with `array<InstanceData>` where:
```wgsl
struct InstanceData {
    model:     mat4x4f,   // current frame
    prevModel: mat4x4f,   // previous frame (for motion vectors)
}
```
Vertex shader indexes via `@builtin(instance_index)`. Motion vectors computed as:
```
prevClip = prevViewProj * prevModel * position
currClip = viewProj * model * position
motionUV = (currClip.xy / currClip.w - prevClip.xy / prevClip.w) * 0.5
```
This is identical to V2's motion vector math, just per-instance instead of per-uniform.

### Physics → Rendering Sync
Each frame:
1. `world.step()` — Rapier advances simulation
2. For each rigid body: read `body.translation()` + `body.rotation()` → build mat4
3. Swap current → previous in instance buffer
4. Write new transforms to instance buffer
5. `device.queue.writeBuffer(instanceBuffer, ...)` — single upload

For ~100 bodies: 100 × 128 bytes (2 mat4s) = 12.8 KB per frame. Negligible.

## Scene Setup

### Domino Chain
- **Dimensions**: 0.1 × 0.5 × 0.05 (half-extents: width × height × depth). Aspect ratio matches real dominoes.
- **Spacing**: 0.15 units apart (1.5× depth = optimal for chain reaction propagation).
- **Arrangement**: Straight line of 30-50 dominoes, with a curve at the end (circular arc).
- **Material**: friction=0.4, restitution=0.1 (low bounce for realistic toppling).
- **Count**: ~50 dominoes total.

### Pendulum
- **Pivot**: Fixed point above the first domino, height ~2 units.
- **Ball**: Sphere, radius 0.15, mass 5 (heavier than dominoes for satisfying impact).
- **Constraint**: `SphericalJoint` connecting fixed body at pivot to dynamic ball. String length ~1.5 units.
- **User control**: Click to release (ball starts held at an angle, release on click). Or: click-drag to aim + release.
- **Visual**: Line from pivot to ball (rendered as a thin cylinder or just a line).

### Floor
- **Static body**: large plane, friction=0.5.
- **Visual**: Same flat quad as V2, textured or colored differently.

### Camera
- **Same FPS camera** as V2 (pointer lock + WASD/EQ).
- **Default position**: Looking at the domino chain from a good angle.

## File Structure

```
web_demo_v3/
├── index.html              # Canvas, toolbar, Rapier CDN script
├── main.js                 # Entry point, game loop, glue
├── physics.js              # Rapier world, domino/pendulum creation, step, transform readout
├── scene.js                # Camera, scene objects registry, instance buffer management
├── renderer.js             # WebGPU device, pipelines, bind groups, dispatch
├── warp.js                 # Noise warp pipeline (extracted from V2 renderer.js)
├── ui.js                   # All buttons, hotkeys, settings persistence, stats overlay
├── shaders.js              # All WGSL shaders (scene, display, warp passes, blue noise)
├── geometry.js             # Vertex data (box, floor, sphere, quad)
└── claude_instructions.md  # Manifest
```

### Module Responsibilities

**`physics.js`** — Owns Rapier world. Exports:
- `initPhysics()` → creates world, ground, dominoes, pendulum
- `stepPhysics()` → advances simulation, returns transform array
- `resetScene()` → re-creates all bodies
- `releasePendulum()` → removes hold constraint, lets ball swing
- Physics parameters (friction, restitution, gravity) configurable

**`scene.js`** — Manages the connection between physics and rendering. Exports:
- `SceneManager` class: tracks all renderable objects, their physics body handles, and instance data
- Handles previous-frame transform storage for motion vectors
- Builds the instance storage buffer each frame
- Camera class (extracted from V2 main.js)

**`renderer.js`** — WebGPU device and scene rendering only. Exports:
- `Renderer` class: device init, scene render pipeline (instanced MRT), display pipeline
- Does NOT contain warp logic (that's in warp.js)

**`warp.js`** — Noise warp compute pipeline extracted from V2. Exports:
- `WarpPipeline` class: all 5 warp compute passes + blue noise
- Takes a WebGPU device and shared textures/buffers as input
- Identical algorithm to V2, just decoupled from scene rendering

**`ui.js`** — All UI logic. Exports:
- `UIManager` class: button creation, hotkey handling, settings persistence
- Event dispatching to other modules
- Stats overlay updates

**`main.js`** — Glue. Imports everything, wires modules together, runs the game loop:
```
init: physics → scene → renderer → warp → ui
loop: physics.step → scene.updateTransforms → renderer.renderScene → warp.run → renderer.display
```

### Why This Split
V2 has 3 files totaling ~1500 lines where renderer.js does everything (scene render, warp, blue noise, profiling, buffer management). V3 splits along clear responsibility boundaries:
- **physics.js** knows nothing about rendering
- **warp.js** knows nothing about the scene (just needs motion texture + noise buffer)
- **renderer.js** knows nothing about physics or noise
- **ui.js** knows nothing about GPU internals
- **scene.js** bridges physics → rendering

## Rapier Integration

### Loading
```html
<!-- CDN, no npm needed -->
<script src="https://cdn.jsdelivr.net/npm/@dimforge/rapier3d-compat@0.14.0/rapier.js"></script>
```
Or ESM import:
```javascript
import RAPIER from 'https://cdn.jsdelivr.net/npm/@dimforge/rapier3d-compat@0.14.0/+esm';
await RAPIER.init();
```

### Domino Creation
```javascript
for (let i = 0; i < NUM_DOMINOES; i++) {
    const rbDesc = RAPIER.RigidBodyDesc.dynamic()
        .setTranslation(i * SPACING, HALF_HEIGHT, 0);
    const rb = world.createRigidBody(rbDesc);
    const colDesc = RAPIER.ColliderDesc.cuboid(0.1, 0.5, 0.05)
        .setFriction(0.4)
        .setRestitution(0.1);
    world.createCollider(colDesc, rb);
}
```

### Pendulum
```javascript
// Fixed pivot
const pivotDesc = RAPIER.RigidBodyDesc.fixed().setTranslation(0, 3, 0);
const pivot = world.createRigidBody(pivotDesc);

// Ball
const ballDesc = RAPIER.RigidBodyDesc.dynamic()
    .setTranslation(-1.5, 1.5, 0);  // held at angle
const ball = world.createRigidBody(ballDesc);
world.createCollider(RAPIER.ColliderDesc.ball(0.15).setDensity(20), ball);

// Joint
const joint = world.createImpulseJoint(
    RAPIER.JointData.spherical(
        { x: 0, y: 0, z: 0 },     // anchor on pivot
        { x: 1.5, y: 1.5, z: 0 }  // anchor on ball (offset = string length vector)
    ),
    pivot, ball, true
);
```

### Transform Readout
```javascript
function readTransforms(bodies, buffer) {
    for (let i = 0; i < bodies.length; i++) {
        const pos = bodies[i].translation();
        const rot = bodies[i].rotation();  // quaternion {x,y,z,w}
        // Build mat4 from pos + quat, write to buffer at offset i*128
        mat4.fromRotationTranslation(tempMat, [rot.x, rot.y, rot.z, rot.w], [pos.x, pos.y, pos.z]);
        buffer.set(tempMat, i * 32);       // current model (16 floats)
        // previous model already at i*32 + 16 from last frame
    }
}
```

## Performance Budget
- Rapier step (100 bodies): <1ms (WASM, trivial workload)
- Transform upload: <0.1ms (12.8KB writeBuffer)
- Scene render (instanced): <1ms (100 instances, simple geometry)
- Warp pipeline: ~12ms at 2048² (same as V2, unchanged)
- Total: ~14ms (~70fps at 2048²), dominated by warp as before

---

## Risk: Motion Vector Quality with Many Objects
V2's single cube produces smooth, predictable motion vectors. With 50+ dominoes toppling:
- Motion vectors at domino edges may be noisy (adjacent pixels belong to different objects)
- Disocclusion will be more frequent (dominoes reveal floor behind them)
- The warp handles this via fresh noise injection at disoccluded pixels (pixelArea=0)
- May need to increase MAX_TICKETS if colliding dominoes create complex motion fields

Mitigation: Start with 20 dominoes, profile ticket usage, scale up.
